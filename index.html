Self-Improving Agent Harnesses

A mutator — an LLM agent (Claude Code) — edits the harness (system prompt, tools, control loop; not the weights) of a separate, frozen legal agent (Harvey, GLM-5.2), to make it better or cheaper. This site indexes every run: what the mutator changed per iteration, the measured train/val scores, and a link to each run's artifacts.

benchmark
Harvey LAB — github.com/harveyai/harvey-labs (MIT) · graded by an ensemble judge (GPT-5-mini + GPT-5 → Sonnet on disagreement)
splits
train = the mutator's workbench · dev = held-out commit gate (ensemble-graded) · val = the invisible verdict the mutator never sees
noise
GLM-5.2 re-rolls the same harness ±0.013 on held-out val (run-to-run variance).

Techniques

The mechanisms built around the mutator — each shown as what it replaced, what it became, and the failure or constraint that motivated it.

New — under-exploration hardeningAdded after the runs above, in response to eff_round1 iter 4 — which committed an unvalidated edit at $9 / 19 min of a $160 / ~3 h budget and overfit. Under-exploration guard: a host-side backstop that, if the mutator writes its done-sentinel while >50% of its experiment budget and ≥1 dev-gate of wall-clock remain unspent, treats it like a park and re-runs it (bounded, preserving committed work) with a “you stopped too early — pivot to a new lever” note — turning the soft CLAUDE.md “don't stop early” rule into a hard stop. Soft budget reminder + adaptive gate estimate: the in-container budget tool now reports both its wall-clock soft deadline and its dollar cap, and the “room for N more dev gates” estimate is measured from the last full gate's duration (the dev giants run ~45–50 min, not a fixed 50-min guess). None of the three runs above used either guard.
Guidance — what the mutator is told
Ambition rewrite of CLAUDE.md
beforeTerse, cautious instructions; the mutator made small, cheap edits and stopped early, under-using its budget.
→
after“Exhaust your money and time. Make foundational, diverse moves. Build your own tools. Reserve budget to validate.” Plus a from-scratch framing.
Why: Early runs under-used their budget and converged on trivial edits.
Efficiency-only objective
beforeMaximize pass-rate (open-ended).
→
afterObjective = cut the carried-input cost (Harvey re-sends everything it has read every turn), with accuracy as a hard gate. Attack the re-carried context specifically.
Why: Harvey is input-bound (≈98% input tokens), so cost is the dominant tractable lever.
Diverse levers + anti-plateau nudge
beforeA menu of directions existed, but the mutator hill-climbed the one lever it had already committed.
→
after“Read your memory first; if a lever is a tapped-out local optimum, pivot to a different family — don't stop at no-change.” (This is what made eff iter 4 try argument-eviction.)
Why: eff iters 1–3 all tuned the same eviction lever; the loop needed a cue to diversify.
Eval signal — what the mutator decides against
Non-overlapping held-out DEV set + val isolation
beforeThe mutator could tune against the same tasks it was scored on, which overfits.
→
afterThree disjoint pools: TRAIN (explore freely), DEV (the commit gate, ensemble-graded), VAL (the verdict the mutator never sees). Val isolation is structural, not honor-system.
Why: A skill-only v0 mutator demonstrably overfit to the visible tasks; a held-out gate is what catches it.
Per-task THRASH signal (+ loop-gating)
beforeOnly the mean cost was reported — a mean cut can sit on top of one task that tripled.
→
afterPer-task turns/tokens ratio vs bare, flagging a re-fetch loop (turns ≥2× with tokens ≥1.5×, or tokens ≥3× with turns ≥1.5×) as a hard reject; a single-axis rise is a watch.
Why: eff iter 1's −21% mean hid a 4.4× per-task thrasher; the flag caught it and forced the iter-2 fix.
Runaway / quality canary (pre-gate abort)
beforeA broken harness edit could run the full (slow, costly) dev gate before its breakage was detected.
→
afterA fast canary on a few tasks runs first; if the edit blows up turns (≥3×) or collapses pass-rate, the iteration is aborted before the dev gate, and the broken edit is never committed.
Why: v1_cont_work iter 2 broke the agent (canary pass-rate 0.11); the canary caught it across retries and aborted instead of committing.
Loop mechanism — state & memory
Carry the whole filesystem forward
beforeEach iteration started from scratch.
→
afterA continuity loop: the mutator's entire workspace (harness + tools + scratch notes + prototypes) carries to the next iteration, so edits and reasoning compound.
Why: One-shot mutators can't build on their own prior work; compounding needs persistent state.
Mutator-written lessons + host-side fallback
beforeNothing was persisted between iterations; one GLM run wrote zero lessons.
→
afterEach iteration ends by writing a .claude/memory lesson, and a host-side fallback writes a factual breadcrumb if the mutator doesn't, so the 'read your memory first' trail never breaks. (The fallback fired on eff iter 4.)
Why: The next iteration's diversification depends on knowing what's already been tried.
Budget & time
Higher per-iteration budget
before$5–6 per run (the v0 trials) — barely enough to read, let alone experiment.
→
afterPer-iteration experiment cap raised to $50 → $160, enough to run real train experiments and a full held-out dev gate.
Why: A trustworthy generalization signal costs money; under-funding it forces commits on noise.
Time + budget reminders
beforeA silent hard timeout — the mutator could start a 90-minute dev gate it couldn't finish, then get killed.
→
afterA wall-clock soft deadline (MUTATOR_DEADLINE_EPOCH), a budget tool reporting both dollars and time (OK / LOW / CRITICAL), and a dev-gate guard that refuses a gate it can't finish.
Why: A run started a gate it couldn't finish and was hard-killed mid-refinement.
Mutator brain
Swappable mutator model (GLM → Opus)
beforeGLM-5.2 only (via a LiteLLM shim).
→
after--mutator-endpoint {glm, opus} at effort xhigh — the rollouts stay GLM, only the mutator brain changes.
Why: To separate 'is the loop good?' from 'is the mutator model good?'.
